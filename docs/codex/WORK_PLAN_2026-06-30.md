# Synapse Work Plan - 2026-06-30

This plan compares the current public repository with:

- `Synapse v6.6设计文档-GPT优化迭代.docx`
- `Codex待办清单_v2026-06-30.docx`
- `Synapse待融合清单_v2026-06-30.docx`

It intentionally keeps private/internal source documents out of the public
repository.

- Architecture language: Taiheng / 太衡, Zhishu / 智枢, Xingtai / 行台, Baigong / 百工

## Phase 0 - Clean Baseline

Goal: make the repo safe for continued iterative development.

Tasks:

1. Reconcile local/remote Git state after API fallback.
2. Commit or discard collaboration baseline docs and ignore-rule changes.
3. Keep internal DOCX design files outside the public repository.
4. Confirm baseline checks:
   - `npm.cmd run i18n:check`
   - `npm.cmd run build`
   - `npm.cmd run preflight:static`

Exit criteria: clean working tree or intentionally staged review set.

## Phase 1 - Secret Guard And Repository Trust

Goal: implement the highest-value Codex backlog item without enabling risky
automation.

Minimum scope:

1. Add a read-only Secret Guard scanner script/domain for staged/source files.
   Initial CLI/static-preflight coverage is implemented by
   `scripts/secret-guard.mjs`; GitHub Actions also runs `secret:scan` before
   static preflight. UI/domain surfacing remains pending.
2. Detect common token/key patterns and forbidden local secret files.
3. Add repository trust preview fields:
   - known workspace
   - dirty workspace
   - untracked sensitive files
   - public/private unknown
   Initial Agent Harness workspace trust preview is implemented; sensitive-file
   trust scoring and public/private classification still need refinement.
4. Surface results in Production Readiness/Security Center as preview-only.

Exit criteria: scanner runs locally, blocks nothing by default, and preflight can
report findings.

## Phase 2 - Agent Safety Execution Layer

Goal: turn V6.6 Agent safety principles into a reusable Taiheng gate.

Minimum scope:

1. Add command allowlist/denylist classification.
   Initial Agent input marker classification is implemented in Agent Harness.
2. Detect dangerous command categories: destructive file ops, credential reads,
   network exfiltration, shell chaining, package install, git rewrite, process
   kill.
3. Reuse existing Agent Harness receipts/audit records.
4. Keep direct execution disabled or explicitly approval-gated.

Exit criteria: Agent Harness can preview safety decisions before execution.

## Phase 3 - Data Source Registry And Project Radar

Goal: evolve data-source preview toward Baigong governance without live-risky
fetches.

Minimum scope:

1. Extend registry metadata for auth, proxy, rate limit, health check, storage
   policy, owner module, and sharing policy.
   Initial registry metadata now also includes adapter kind, observation policy,
   and freshness policy.
2. Add read-only Project Radar descriptors:
   - GitHub Trending
   - OSSInsight
   - Hugging Face Trending
   Initial disabled/quarantined descriptors are implemented.
3. Keep observations quarantined and credential-free.
4. Add Baigong module manifest guidance for capabilities, data sources,
   permissions, and Zhishu admission policy.
   Initial public template is implemented in `docs/BAIGONG_MODULE_MANIFEST.md`.

Exit criteria: radar sources appear as registered preview sources with health
and policy metadata.

## Phase 4 - Bilingual Completion And UI Maintainability

Goal: make bilingual usage real across the main product surface.

Minimum scope:

1. Expand `src/i18n/translations.ts` coverage panel by panel.
2. Add UI smoke checks for language switching.
3. Extract one domain hook from `src/App.tsx` at a time, beginning with the
   safest read-only preview domain.

Exit criteria: no new user-facing text is added without English and Simplified
Chinese keys.

## Phase 5 - Release 0.0.1

Goal: ship a bilingual installer only after the previous phases are stable.

Tasks:

1. Bump public version to `0.0.1`.
2. Run release checks and MSI build.
3. Generate SHA-256 sidecar.
4. Publish GitHub Release with guarded baseline notes.

Exit criteria: downloadable Windows MSI with release notes that do not overclaim
automation capability.
