# Synapse 下一步开发规划

版本：2026-06-25 重排版

依据：

- `Synapse v6.4 设计文档-GPT优化迭代.docx`
- `Codex_下一步开发建议.md`
- 当前 Tauri + Rust + React 实现

## 1. 总体结论

当前项目不应改建为独立的 FastAPI 后端，也不应重新搭建一套平行骨架。
现有 Tauri/Rust 后端已经具备智枢、任务中心、军火库、安全策略和本地存储的
可运行基础，下一阶段应沿现有架构补齐可靠闭环。

下一阶段主线调整为：

```text
稳定数据契约
  -> 快照与回滚
  -> 统一审计和证据链
  -> 任务状态机
  -> 后台调度
  -> 受控适配器执行
  -> 信息聚合与来源置信
  -> 智枢检索和自生长
```

每日简报、A 股策略实验室、邮件模板等属于领域能力，应通过任务模板和军火库
适配器接入，不进入核心模型硬编码。

## 2. 架构决策

### 2.1 保留当前技术栈

- 桌面壳和本地运行时：Tauri 2
- 核心服务和安全边界：Rust
- 工作台：React + TypeScript
- 当前 JSON 存储：作为原型兼容层继续保留
- 中期结构化存储：在仓储接口稳定后迁移到 SQLite
- Python：仅作为军火库脚本运行时，不作为第二套核心后端

### 2.2 核心与领域插件分离

核心只保存通用概念：

- 智枢条目、来源、置信、审核和层级
- 任务、计划、运行、审批和证据
- 工具适配器、权限、健康状态和调用记录
- 快照、回滚和审计事件

领域插件保存专用概念：

- 每日简报模板和信息源组合
- A 股因子、模拟账户、策略版本和评估指标
- 司法鉴定工作流
- 写作、电脑维护和开发辅助流程

### 2.3 安全能力前置

设计文档原 Phase 5 的安全护栏必须前移。任何后台任务、网络访问、脚本、
Agent、本地应用、邮件或文件修改在接入真实执行前，必须先具备：

- 明确的权限分级
- 预览和审批
- 快照或补偿策略
- 持久审计记录
- 幂等和重复执行保护
- 凭证隔离

## 3. 当前成熟度判断

### 已有基础

- 智枢 L0/L1/L2、分类元数据、候选审核和拒绝隔离
- 任务方向、候选挖掘、运行请求、审批和本地候选深化
- 手动调度 tick、周期预览和运行去重
- 军火库注册表、PATH 探测、允许状态和输出入库策略
- 权限策略、信息聚合门控、执行契约和系统能力状态
- JSON envelope 与未来 schema 版本拒绝机制

### 主要缺口

- 没有正式的数据迁移和恢复流程
- 没有通用对象快照与一键回滚
- 审计、审批、执行和入库记录尚未形成统一事件链
- 后台调度仍未启动
- 军火库大部分仍是预览，不执行真实适配器
- 信息聚合没有真实抓取、多源交叉验证和来源健康管理
- 智枢检索仍偏关键词匹配，缺少统一检索和关系索引
- 领域模块边界尚未代码化

## 4. 分阶段路线

## Phase 0：存储可靠性收口

目标：先保证现有数据可升级、可恢复、不会静默损坏。

任务：

1. 完成 JSON schema 版本测试和迁移入口。
2. 增加原子写入：临时文件、同步、替换、失败保留旧文件。
3. 增加损坏文件隔离与可读错误报告。
4. 为旧数组格式和 schema v0 增加兼容测试。
5. 定义 `StoreRepository` 边界，为 SQLite 迁移做准备。

验收：

- 新版本拒绝读取未知未来 schema。
- 写入中断不破坏最后一个有效版本。
- 旧数据可读并能显式升级。
- store 定向测试全部通过。

## Phase 1：快照、回滚与统一审计

目标：建立一条可持久化、可追踪、可恢复的安全底座。

任务：

1. 定义统一 `AuditEvent`：
   `actor/action/target/risk/decision/input_hash/result/error/timestamp`。
2. 定义 `SnapshotRecord` 和对象版本号。
3. 对智枢审核、规则变更、任务审批、工具允许状态变更创建审计事件。
4. 对高风险写入和状态变更先创建快照。
5. 实现快照列表、预览和回滚命令。
6. 在安全中心展示审计链和恢复点。

验收：

- 任一受保护对象可查看变更前后记录。
- 回滚不会覆盖审计历史。
- 重复回滚具有明确、可预测的结果。

## Phase 2：任务中心正式状态机

目标：把当前多个预览状态统一为一个明确的任务生命周期。

推荐状态：

```text
captured -> classified -> planned -> awaiting_approval
-> approved -> running -> succeeded/failed/cancelled -> archived
```

任务：

1. 区分 `TaskDefinition`、`TaskRun`、`TaskStep`、`TaskArtifact`。
2. 将证据、审批、执行结果和智枢入库候选关联到同一 `run_id`。
3. 为每个状态转换定义前置条件和审计事件。
4. 增加失败重试、取消、超时和幂等键。
5. 保留现有方向和候选模型，作为任务定义和任务产物的一部分迁移。

验收：

- 一个本地任务可从创建走到归档。
- 非法状态跳转被拒绝并记录原因。
- 相同幂等键不会重复执行。

当前进度：已完成第一步，将旧审批/执行字符串组合归一为持久生命周期状态，
并集中校验状态转换；已增加运行中、成功和失败状态及时间/错误摘要，取消与归档
状态及工作台操作也已完成。下一步收口幂等键和任务产物关联。

幂等键现已完成：调度运行按方向和周期时间桶去重，候选深化按源候选永久复用，
旧记录读取时生成稳定 legacy 键。下一步建立独立任务产物记录。

独立任务产物记录现已完成，候选输出可按运行查询，产物索引失败会补偿本次候选。
Phase 2 的状态、幂等、运行结果和产物主链已形成，下一阶段进入安全后台调度。

## Phase 3：安全后台调度

目标：在手动 tick 已可靠的基础上启用真正的本地后台循环。

任务：

1. 增加单实例 scheduler 和启动/停止生命周期。
2. 持久化下次运行时间、最近成功时间和失败退避。
3. 使用任务幂等键防止重复触发。
4. 重启后恢复未完成运行，不自动越过审批。
5. 增加 missed-run、重试和手动补跑策略。

验收：

- 重启应用不会产生重复运行。
- 未审批任务不会因定时器启动而执行。
- 调度失败可以在工作台定位并重试。

当前进度：已完成持久化 scheduler state、单实例租约、心跳、过期接管和 tick
健康记录；受配置控制的后台循环、退出停止和租约释放也已完成。下一批补失败
退避和重启运行恢复。

失败指数退避、退避期间续租、租约丢失停机和遗留 `running` 恢复均已完成。
Phase 3 安全后台调度主链完成，下一阶段进入军火库适配器最小执行闭环。

## Phase 4：军火库适配器最小执行闭环

目标：先用 mock 和低风险本地工具打通真实适配器协议。

任务：

1. 定义稳定的 adapter manifest、输入输出 schema 和健康检查协议。
2. 实现 `mock_cli` 的 dry-run 和受控 execute。
3. 执行前经过策略、允许列表、路径和参数校验。
4. 执行后写入审计、耗时、退出码、摘要和产物引用。
5. 增加超时、取消、输出大小限制和敏感信息清洗。
6. 第二个适配器优先选择只读 Python 工具或只读本地命令。

验收：

- mock 适配器可完整执行并回写任务运行。
- 未允许、未审批或风险不匹配的调用无法启动。
- 失败不会直接污染智枢。

当前进度：内置 `mock-cli` 已完成 dry-run、允许列表、Task Run 审批、显式执行
审批、结构化回执、持久审计和 Task Artifact 闭环；真实外部进程仍全部禁用。

## Phase 5：信息聚合与来源置信

目标：实现项目唯一的受控联网入口。

任务：

1. 定义 `SourceDescriptor`、`SourceObservation`、`ConfidenceAssessment`。
2. 实现来源白名单、健康状态、时效和字段覆盖记录。
3. 先接入静态 fixture，再接入一个真实只读 HTTP 来源。
4. 对高影响结论支持多源交叉验证和冲突标记。
5. 外部内容进入隔离区，提示注入检测后才能参与计划。
6. 聚合结果只生成智枢候选，不直接持久晋升。

验收：

- 每条外部结论可追溯到来源和抓取时间。
- 来源冲突不会被静默合并。
- 网络失败能降级为缓存、手动导入或不执行。

当前进度：已完成离线 fixture 多源观察、时效/覆盖率/冲突评估和置信评分，
所有结果保持隔离且不写智枢；观察持久化、来源筛选和工作台健康历史也已完成。
受限 JSON/CSV 手动导入降级通道也已完成，不读取任意文件路径。下一步评估并
接入一个真实只读 HTTP 来源。

首个配置白名单 HTTP JSON 来源已完成，包含 GET-only、HTTPS、无重定向、无凭证、
超时、响应大小、内容类型、隔离、观察历史和审计限制。Phase 5 的单来源闭环完成。

## Phase 6：智枢检索与自生长

目标：把“保存内容”升级为“可靠检索、关联、总结和沉淀”。

任务：

1. 建立统一检索接口，组合类型、标签、层级、置信、时效和全文条件。
2. 增加关系记录，保存关联原因和证据，不直接修改原条目。
3. 增加定期总结、重复压缩、过期检测和冲突发现。
4. 将成功经验、失败规避和规则候选统一进入审核队列。
5. 为 SQLite 迁移实现仓储适配器，并保留 JSON 导入/导出。

验收：

- 检索结果能解释命中原因。
- 自动总结和关联都必须可审核、可拒绝。
- 拒绝内容不会继续参与推荐和自生长。

当前进度：统一智枢检索、组合过滤、置信/时效条件、命中解释、独立关系索引、
关系候选去重与接受/拒绝审核已完成；原智枢条目不会被关系发现流程直接修改。
重复、过期和保守冲突检测也已进入独立维护队列，支持去重和接受/拒绝审核，
不会自动合并、删除或改写原智枢条目。
智枢记忆、关系和维护集合已迁入 SQLite 仓储边界，旧 JSON 首次自动导入且保留，
并提供版本化 JSON bundle 的显式导出与事务导入。

## Phase 7：领域插件试点

目标：验证核心能承载真实场景，而不污染底座。

推荐顺序：

1. 每日简报：验证调度、聚合、模板、归档和通知。
2. 电脑助手只读诊断：验证本地适配器和安全报告。
3. A 股策略实验室：验证结构化数据、策略版本和回滚。

A 股模块必须独立为 `quant_lab` 领域包，默认只做研究和模拟。数据不足时
降级为盘后确认或停止生成交易动作，不在核心代码中写死股票、账户和邮箱。

当前进度：每日简报领域插件已完成模板预览、来源置信门控、隔离观察持久化、
已审批 Task Run 归档和 Task Artifact 输出；外部通知仍保持禁用。
电脑助手只读诊断已完成运行环境、临时目录、PATH、智能体发现和配置警告检查，
支持经已审批 Task Run 归档报告，全程不启动进程或修改系统。
A 股策略实验室已独立为 `quant_lab`，支持受限历史 CSV、版本化均线模拟、
数据不足停止、风险声明和经已审批 Task Run 归档；不连接券商或生成实盘委托。

## Phase 8：多 Agent、浏览器与多端

只有在 Phase 0-7 稳定后再推进：

- Agent Harness 原生/深度双模式
- 线性团队和圆桌团队
- 浏览器自动化
- 本地应用调用
- 邮件、飞书、微信推送
- 多端轻客户端和可选云中转

这些能力共享同一任务状态机、安全网关、审计链和入库门控，不建立旁路。

当前进度：Agent Harness 第一阶段已完成 Codex、Claude Code、Gemini、Hermes
本地探测复用、原生/深度双模式契约、智枢上下文隔离、参数向量预览、四层就绪
门控和持久审计；尚未启动任何真实外部进程。
首个真实适配器已限定为 Codex CLI：仅在探测、允许、Task Run 审批和二次执行确认
全部满足后，以 ephemeral/read-only/忽略用户执行配置的方式运行，带 120 秒超时、
256 KB 输出上限和隔离 Task Artifact；其他智能体仍保持 dry-run。
浏览器自动化首阶段已复用本机 Python Playwright/Chromium，采用精确主机白名单、
逐请求拦截、双工具允许、Task Run 审批和二次确认，仅开放导航、标题、受限正文与
截图，结果进入隔离 Task Artifact，不开放点击、表单、凭据、上传下载或任意脚本。
Agent 团队第一阶段已支持 2-4 成员、1-3 轮的线性交接与圆桌汇总执行图，展示调用
预算并强制成员输出隔离；当前仅预览，不启动多智能体进程。
本地应用桥接首阶段已限定为代码内规范化的 Windows 记事本描述符，工具允许与应用
允许分离，要求 Task Run 审批和二次确认，仅无参数启动并记录回执，不读取凭据、
应用配置或窗口内容，也不把“已启动”视为任务完成。
统一通知网关已建立 email/飞书/微信通道契约，SMTP 邮件支持配置、环境变量凭据、
Task Run 通道许可、二次确认和投递回执；凭据不进入 JSON、审计或产物，飞书和微信
仍保持不可执行的适配器契约。

## 5. 立即执行的开发批次

建议从以下四个小阶段开始：

### Batch A：Schema 兼容与原子写入

- 补 schema v0、旧数组和未来版本测试。
- 实现安全原子写入。
- 更新存储格式说明。

### Batch B：快照数据模型

- 已新增快照 store。
- 已支持智枢条目、任务方向和军火库允许状态的手动快照与查询。
- 自动变更前快照和回滚仍等待统一审计事件完成。

### Batch C：统一审计事件

- 已新增统一持久化审计事件和筛选查询接口。
- 智枢审核、任务运行审批、任务候选审核和军火库允许状态变更已写入事件。
- 旧 review/history 保持兼容；跨文件事务补偿将在 Batch D 收紧。

### Batch D：首个可回滚流程

- 已选择“智枢候选审核”作为首个完整对象。
- 已完成变更前快照、审核、审计、失败补偿和回滚前保护。
- 工作台已提供最近智枢恢复点与最小恢复入口。

完成 Batch D 后，再决定是否立即引入 SQLite。若 JSON 文件数量、跨对象查询
或事务需求已经明显增加，则进入 SQLite；否则先完成任务状态机再迁移。

## 6. 暂缓事项

- FastAPI 平行后端
- 复杂多 Agent 团队
- 浏览器写操作和本地应用控制
- 自动邮件和多端推送
- 实时盘中交易模拟
- UI 动画和品牌细节重做
- 未经审核的自动智枢写入

## 7. 每阶段通用完成标准

每个阶段必须同时满足：

1. 模型和状态转换有定向单元测试。
2. 高风险路径有拒绝测试。
3. 持久数据保持向后兼容或提供显式迁移。
4. UI 不宣称尚未真正可用的能力。
5. 文档同步记录当前能力和限制。
6. 仅在阶段边界运行完整 Rust 测试和前端构建。
